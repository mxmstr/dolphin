
### Summary of Stereoscopic Rendering Implementation in Dolphin VR

The implementation of stereoscopic 3D rendering in this fork is a multi-faceted approach, deeply integrated into the emulator's core video pipeline. It combines direct VR SDK integration for camera properties, a sophisticated graphics command replay system for performance, and extensive configuration options.

The key files modified to achieve this are:

1.  **`Source/Core/VideoCommon/VR.cpp`**: The central VR abstraction layer.
2.  **`Source/Core/VideoCommon/OpcodeDecoding.cpp`**: The core graphics command interpreter.
3.  **`Source/Core/VideoCommon/VRGameMatrices.cpp`**: Handles view matrix calculations with head tracking.
4.  **`Data/Sys/Shaders/lens_distortion.glsl`**: A post-processing shader for lens correction.
5.  **`Source/Core/VideoCommon/VideoConfig.h` & `VideoConfig.cpp`**: Configuration management.
6.  **`Source/Core/VideoCommon/VROpenVR.h`**: Header for OpenVR SDK integration.

---

### 1. `VR.cpp` and `VROpenVR.h`: VR SDK Integration and Camera Control

These files are responsible for interfacing with the VR runtimes (like OpenVR) to get the necessary parameters for rendering two distinct views.

*   **Projection Matrix Generation (`VR_GetProjectionMatrices`)**:
    *   This function is the heart of the stereoscopic setup. It queries the active VR SDK (e.g., OpenVR via the handle in `VROpenVR.h`) for the projection matrix of each eye.
    *   For OpenVR, it calls `m_pHMD->GetProjectionMatrix(vr::Eye_Left, ...)` and `m_pHMD->GetProjectionMatrix(vr::Eye_Right, ...)`.
    *   This is critical because HMDs use asymmetric frustums (the view is not centered) to maximize the field of view. The SDK provides the exact matrices needed to render correctly for each eye's physical lens position relative to the screen.

*   **Eye Pose and IPD (`VR_GetEyePos`)**:
    *   This function retrieves the position of each eye relative to the center of the head. This is effectively the Interpupillary Distance (IPD).
    *   For OpenVR, this is derived from `m_pHMD->GetFloatTrackedDeviceProperty(..., vr::Prop_UserIpdMeters_Float, ...)`.
    *   This offset is used to translate the main head-tracked camera position slightly to the left for the left eye's view and to the right for the right eye's view, creating the 3D parallax effect.

*   **Dynamic FOV for Comfort (`VR_NewVRFrame`)**:
    *   The code includes logic to dynamically reduce the Field of View (FOV) as a motion sickness reduction feature.
    *   It does this by directly modifying the `g_eye_fov` tan values, which are then used by `VR_GetProjectionMatrices` to generate a projection with a narrower view. This creates a "tunnel vision" effect during intense motion.

### 2. `VRGameMatrices.cpp`: 6DOF View Matrix Calculation

This file is a prerequisite for stereoscopy, as it calculates the primary "head" camera's position and orientation in the game world before the per-eye offsets are applied.

*   **Combining Head Tracking and Game Camera**: The `CalculateViewMatrix` function takes the 6DOF head tracking data (`g_head_tracking_matrix` and `g_head_tracking_position` from `VR.cpp`) and combines it with the game's own camera matrix (`g_game_camera_rotmat`).
*   **World-Space UI**: It contains extensive logic to transform 2D elements like HUDs into 3D objects positioned at a specific depth (`fHudDistance`). This ensures that UI is not "stuck" to the screen and has a proper sense of depth in stereo, making it comfortable to view.

The final view matrix for each eye is a combination of the matrix from this file and the per-eye translation from `VR_GetEyePos`.

### 3. `OpcodeDecoding.cpp` and `VR.cpp`: Asynchronous Timewarp (Opcode Replay)

This is the most sophisticated part of the implementation, designed to solve the critical performance problem of maintaining high framerates in VR.

*   **Command Recording (`OpcodeDecoding.cpp`)**:
    *   The `Run()` function, which processes the GPU's command FIFO, was modified. When `g_opcode_replay_log_frame` is true, instead of just executing graphics commands, it saves them into a buffer (`timewarp_logentries`).
    *   This effectively records every draw call, state change, and register write for one complete game frame.

*   **Command Replay (`VR.cpp`)**:
    *   Functions like `OpcodeReplayBufferInline()` contain the control logic. Based on the game's actual framerate and the HMD's required refresh rate (e.g., 90Hz), it calculates how many "extra" frames need to be generated.
    *   For each extra frame, it iterates through the recorded `timewarp_logentries` and calls `OpcodeDecoder::Run()` to re-execute them.
    *   Crucially, before replaying the frame, it updates the head tracking data. This means the re-rendered frame uses the very latest head position, even though the game logic hasn't advanced.
    *   This mechanism decouples the rendering framerate from the game's simulation framerate, providing a smooth, low-latency head tracking experience that is essential for comfortable VR. It is a software implementation of what is now commonly known as Asynchronous Timewarp or Reprojection.

### 4. `Data/Sys/Shaders/lens_distortion.glsl`: Post-Processing

This file handles the final step in the stereo rendering pipeline for HMDs that require it.

*   **Barrel Distortion Correction**: The GLSL shader takes the rendered image for an eye and applies an inverse barrel distortion. VR HMDs use lenses that warp the image, so the software must pre-warp it in the opposite direction for the final image to look correct to the user.
*   **Per-Eye Operation**: The shader's logic differs based on the eye (`if (layer == 1)`), applying an `EYE_OFFSET` to correctly center the view for each eye. It also includes adjustable parameters for the amount of distortion and zoom.

### 5. `VideoConfig.h` and `VideoConfig.cpp`: Configuration

These files were heavily extended to expose all the new stereoscopic and VR settings to the user.

*   **New Settings**: Dozens of VR-specific variables were added to the `VideoConfig` struct, such as `iStereoMode`, `bOpcodeReplay`, `fHudDistance`, `bDisable3D`, etc.
*   **Validation (`VerifyValidity`)**: Logic was added to automatically enable stereoscopic mode (`iStereoMode = STEREO_OCULUS`) when a compatible HMD is detected. It also prevents enabling stereo if it conflicts with other settings, like `bUseRealXFB`.
*   This allows for fine-tuning of the stereo effect, comfort options, and performance enhancements on a per-game or global basis.
